Guía del Catálogo UX

Todo lo que necesitas saber para publicar, versionar y gestionar prototipos en el repositorio del equipo UXplorers de Skandia.

0

¿Qué es el catálogo?

El catálogo UX es un repositorio centralizado donde el equipo UXplorers de Skandia guarda todos los prototipos HTML. Funciona como una biblioteca viva: cada prototipo tiene su carpeta, sus versiones y aparece automáticamente en una página web accesible para todo el equipo.

URL del catálogo

Puedes ver todos los prototipos publicados en:
https://uxplorerscolombia.github.io/ux-prototypes/catalog/

¿Cómo funciona por dentro?

Cuando un diseñador agrega un prototipo y hace push al repositorio, GitHub ejecuta automáticamente un proceso que:

  1. 1
    Lee todos los proyectos

    El sistema revisa la carpeta projects/ y encuentra todos los artefactos guardados.

  2. 2
    Genera el catálogo

    Crea automáticamente la página de inicio con tarjetas para cada prototipo, incluyendo nombre, autor, estado y fecha.

  3. 3
    Publica en internet

    La página queda disponible en la URL del catálogo aproximadamente 2 minutos después del push.

No necesitas hacer nada extra

La actualización del catálogo es completamente automática. Solo necesitas hacer push y esperar un par de minutos.

1

Cómo agregar un prototipo

Hay dos formas de agregar un prototipo al catálogo. La primera es la recomendada para el equipo.

Camino A — Recomendado: Agente Claude Code

El agente te guía paso a paso, crea todos los archivos automáticamente y te dice exactamente qué comandos correr. No necesitas saber nada técnico.

Camino A — Con el agente Claude Code (recomendado)

  1. 1
    Abre Claude Code en la terminal

    Navega a la carpeta del repositorio y escribe claude para abrir el agente.

  2. 2
    Dile al agente que guarde el prototipo

    Puedes escribir frases como:

    "Guarda este prototipo" · "Agrega este HTML al repositorio" · "Ya terminé el artefacto, guárdalo"

  3. 3
    Responde las preguntas del agente (una por una)

    El agente te pedirá estos datos:

    📌 Nombre del prototipo — Ej: "Onboarding Digital"

    📌 Tu nombre (autor) — Ej: "Laura Torres"

    📌 Proyecto o iniciativa — Ej: "Fondo de pensiones voluntario"

    📌 Descripción breve — Ej: "Flujo de vinculación para usuarios nuevos"

    📌 Estado — Por defecto: "en revisión"

  4. 4
    Pega el HTML cuando te lo pida

    Si tienes el HTML listo, el agente te pedirá que lo pegues. Si aún no lo tienes, puede crear una carpeta vacía para que la llenes después.

  5. 5
    El agente te dará los 3 comandos finales

    Copia y ejecuta los comandos que el agente te muestra. Eso publica el prototipo en internet.

Camino B — Manual (con script)

Si no tienes Claude Code disponible, puedes usar los scripts incluidos en el repositorio:

Windows

Haz doble clic en nuevo-artefacto.bat en la carpeta raíz del repositorio.

Mac / Linux

Ejecuta ./nuevo-artefacto.sh en la terminal dentro de la carpeta del repositorio.

Ambos scripts son interactivos: te harán las mismas preguntas que el agente y crearán la estructura de archivos automáticamente.

2

Versiones

Cada prototipo puede tener múltiples versiones. Una nueva versión representa una iteración significativa sobre el mismo artefacto — no un archivo diferente, sino una evolución del mismo prototipo.

¿Cuándo crear una nueva versión?

Crear v2 cuando...
  • Hay cambios importantes de flujo o diseño
  • Se incorporó feedback de revisión
  • El equipo aprobó pasar a la siguiente iteración
  • Necesitas mantener la v1 como referencia histórica
No crear v2 cuando...
  • Solo corriges un typo o error menor
  • Es un cambio que nadie necesita rastrear
  • La versión anterior nunca fue compartida

Cómo se nombran las versiones

Las versiones siempre siguen el patrón v1, v2, v3… y se crean de forma secuencial. El sistema detecta automáticamente la versión más alta y crea la siguiente.

Nombres válidos
  • v1
  • v2
  • v3
Nombres inválidos
  • final
  • v2_revisado
  • v1-ok
  • nuevo
Nunca sobrescribas una versión existente

Si v1 ya existe, siempre crea v2. El sistema nunca debe perder el historial de versiones anteriores.

3

Estados del prototipo

Cada prototipo tiene un estado que refleja en qué momento del proceso de diseño se encuentra. El estado aparece como un badge de color en el catálogo.

Estado Badge Significado ¿Cuándo usarlo?
en revisión en revisión El prototipo está siendo evaluado por el equipo o stakeholders. Estado por defecto al subir por primera vez.
en iteración en iteración Se recibió feedback y el diseñador está incorporando cambios. Cuando hay una v2 en progreso basada en comentarios.
aprobado aprobado El prototipo fue validado por el equipo y está listo para la siguiente fase. Después de la aprobación formal del flujo.
handoff handoff El artefacto fue entregado al equipo de desarrollo para implementación. Cuando el equipo técnico ya tiene las especificaciones.
archivado archivado El prototipo ya no está activo pero se conserva como referencia histórica. Proyectos descontinuados o reemplazados por una nueva versión.

¿Cómo cambio el estado?

El estado vive en el archivo meta.json que está en la raíz de cada proyecto. Para cambiarlo, edita el campo "status" con uno de los valores de la tabla, luego haz push. El catálogo se actualizará automáticamente.

Los proyectos archivados siguen visibles

Los prototipos con estado archivado aparecen en el catálogo con un badge gris y se pueden filtrar. No desaparecen del catálogo.

4

Publicar con git

Para que tu prototipo aparezca en el catálogo en internet, necesitas ejecutar 3 comandos en la terminal. Estos comandos le dicen a GitHub que hubo cambios nuevos que debe publicar.

¿Qué es git?

Git es un sistema que lleva el historial de todos los cambios del repositorio. Piénsalo como el "Control + Z" de todo el equipo, pero además permite publicar los cambios en internet.

Los 3 comandos

Abre la terminal, asegúrate de estar en la carpeta del repositorio y ejecuta estos comandos en orden:

Comando 1 de 3 — Marcar los cambios

Terminal 
git add .

Este comando le dice a git: "marca todos los archivos nuevos o modificados para incluirlos en el siguiente guardado".

Comando 2 de 3 — Guardar con un mensaje

Terminal — reemplaza los valores entre [ ] 
git commit -m "feat: [nombre-del-prototipo] [vN] - [Tu Nombre]"

Piensa en esto como el nombre de un archivo cuando lo guardas. Describe qué contiene este cambio.

Ejemplo de mensaje de commit

Si tu prototipo se llama "Onboarding Digital", estás en la versión 1 y tu nombre es Laura Torres, el comando sería:

git commit -m "feat: onboarding-digital v1 - Laura Torres"

Comando 3 de 3 — Publicar en internet

Terminal 
git push

Envía todos los cambios guardados al servidor de GitHub. A partir de aquí, el proceso es automático.

¿Cuánto tiempo tarda en aparecer?

Después de hacer git push, GitHub procesa los cambios automáticamente. En aproximadamente 2 minutos tu prototipo aparecerá en el catálogo y tendrá su propia URL pública.

URL de tu prototipo publicado

Cada prototipo tiene una URL permanente con esta estructura:

URL pública 
https://uxplorerscolombia.github.io/ux-prototypes/projects/[nombre-proyecto]/[vN]/
5

Convenciones

Para que el catálogo funcione correctamente y el equipo pueda encontrar fácilmente cada prototipo, todos los nombres deben seguir las mismas reglas.

Nombres de carpeta: kebab-case

Los nombres de carpeta deben estar en kebab-case: todo en minúsculas, sin espacios, sin acentos, sin caracteres especiales. Los espacios se reemplazan con guión medio -.

Nombre original Carpeta resultante
Onboarding Digital onboarding-digital
Simulador Flujo de Caja simulador-flujo-de-caja
Menú Hub Mobile menu-hub-mobile
Calculadora Ahorro Pensional calculadora-ahorro-pensional
No necesitas calcular esto manualmente

El agente Claude Code y los scripts convierten automáticamente el nombre que das al formato correcto. Solo debes saber esto si editas los archivos directamente.

Estructura obligatoria de archivos

Cada proyecto debe tener exactamente esta estructura. No inventar nombres de archivos ni carpetas adicionales.

Estructura 
# Estructura obligatoria
projects/
└── nombre-del-prototipo/
    ├── meta.json          ← siempre en la raíz
    └── v1/
        └── index.html     ← el artefacto HTML

Formato del mensaje de commit

El mensaje debe seguir siempre este formato exacto:

Formato del commit 
"feat: [nombre-en-kebab-case] [vN] - [Nombre Autor]"
Mensaje ¿Válido?
feat: onboarding-digital v1 - Laura Torres Sí
feat: simulador-ahorro v2 - Javi Sí
subiendo prototipo No
actualizaciones No
6

Preguntas frecuentes

¿Qué pasa si hago push y el catálogo no se actualiza?

Espera al menos 2 minutos. Si después de 5 minutos el catálogo sigue sin actualizarse, ve a la página del repositorio en GitHub, haz clic en la pestaña "Actions" y verifica si hay algún error en el proceso automático. Comparte la pantalla con alguien del equipo técnico si no entiendes el error.

¿Puedo subir imágenes o archivos adicionales junto al HTML?

Sí, puedes agregar imágenes u otros archivos dentro de la carpeta de la versión (ej: v1/imagen.png). El HTML puede referenciarlos con rutas relativas como <img src="imagen.png">. Sin embargo, para mantener los prototipos lo más ligeros posible, se recomienda que el HTML sea autocontenido (con imágenes embebidas en base64 o usando URLs externas).

¿Qué hago si cometí un error en el nombre de la carpeta?

Si el error está solo en tu máquina y aún no hiciste push, puedes renombrar la carpeta directamente. Si ya hiciste push, lo más simple es crear una nueva carpeta con el nombre correcto, mover los archivos y hacer un nuevo push. Contacta a alguien del equipo técnico si no sabes cómo hacerlo sin perder los archivos.

¿Puedo editar el meta.json después de publicar?

Sí. Puedes abrir el archivo meta.json con cualquier editor de texto (Bloc de notas, VS Code, etc.), cambiar los campos que necesites (nombre, descripción, estado, autor), guardar el archivo y luego hacer los 3 comandos de git (add, commit, push). El catálogo reflejará los cambios en unos minutos.

¿Cómo cambio el estado de un prototipo a "aprobado"?

Abre el archivo projects/[nombre-proyecto]/meta.json en un editor de texto. Busca la línea que dice "status": "en revisión" y cámbiala a "status": "aprobado". Guarda el archivo y haz los 3 comandos de git para publicar el cambio.

¿El catálogo tiene un buscador o filtros?

Sí. El catálogo tiene filtros por autor y por estado. Puedes combinarlos para encontrar, por ejemplo, todos los prototipos de un autor específico que están en estado "aprobado". Los filtros aparecen en la barra superior del catálogo.

¿Puedo compartir el enlace del prototipo con alguien externo al equipo?

Sí. Las URLs de los prototipos son públicas: cualquier persona con el enlace puede verlas sin necesitar cuenta en GitHub. La URL tiene la forma https://uxplorerscolombia.github.io/ux-prototypes/projects/[nombre-proyecto]/[vN]/. Puedes compartirla por correo, Slack o cualquier canal.

¿Qué pasa si borro accidentalmente un archivo del repositorio?

Gracias a git, todo tiene historial. Nada se pierde permanentemente a menos que alguien haga una operación destructiva deliberada. Si borraste algo accidentalmente, contacta a alguien del equipo técnico antes de hacer push — es más fácil recuperarlo antes de publicar el error.
